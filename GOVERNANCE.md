# Dure 거버넌스

이 문서는 공개 저장소의 의사결정, 책임 위임 및 긴급 대응 원칙을 정의합니다. 실제 권한은 GitHub 조직·저장소 설정, 보호 규칙, 배포 환경 권한 및 서명 키 보관 정책으로 집행합니다.

## 역할

개인의 이름을 이 문서에 고정하지 않습니다. 각 역할의 담당자는 GitHub 조직 또는 저장소의 권한 설정과 해당 릴리스 기록에서 확인합니다.

| 역할 | 책임 | 권한 원칙 |
| --- | --- | --- |
| 저장소 Maintainer | PR triage, 보호 브랜치 관리, 일반 변경 승인 | `main` 직접 변경·force push를 하지 않고 보호 규칙을 따릅니다. |
| Release Manager | 버전 승인, tag·공식 release·APT 게시 판단 | [릴리스 거버넌스](dure/docs/release-governance.md)와 [릴리스 실행 체크리스트](dure/docs/release-runbook.md)를 따릅니다. |
| Security Triage 담당자 | 취약점 비공개 접수, 영향 평가, 공개 시점 조정 | 공개 Issue가 아닌 [보안 정책](SECURITY.md)의 비공개 경로를 우선 사용합니다. |
| 문서 담당자 | 운영 계약·보안 경계·사용자 안내의 최신성 검토 | CLI·API·설정·수명주기 변경 PR에서 문서 영향을 확인합니다. |
| 행동강령 대응 담당자 | 신고 접수, 보호 조치, 기록 보관 | [행동강령](CODE_OF_CONDUCT.md)의 비공개 연락 경로와 절차를 사용합니다. |

한 사람이 여러 역할을 맡을 수 있지만, 자신의 릴리스 승인·보안 영향 판단·행동강령 사건을 단독으로 종결하지 않도록 가능한 범위에서 다른 Maintainer의 검토를 받습니다.

## 의사결정과 승인

- 일반 변경은 Issue 또는 PR에 문제, 범위, 검증 결과와 운영 영향을 남깁니다. 보호 브랜치 규칙이 요구하는 검토·CI를 통과해야 합니다.
- API, DB migration, Agent task enum, 인증·권한, 배포 경계, 패키지·서명 키, 공개 네트워크 노출을 바꾸는 변경은 관련 역할의 명시적 검토를 받습니다.
- 의견이 갈리면 안전성, 되돌릴 수 있음, 문서화 가능성, 사용자 영향 순으로 판단합니다. 합의하지 못하면 위험이 더 낮은 선택 또는 보류를 택합니다.
- 결정과 예외는 PR, Issue 또는 릴리스 기록에 남겨 나중에 검토할 수 있게 합니다. 개인 credential이나 보안 세부사항은 기록하지 않습니다.

## 대리와 부재 대응

- 담당자가 부재이면 다른 권한 있는 Maintainer가 임시 대리자를 지정하고, 범위·기간·결정을 기록합니다.
- Release Manager 대리는 production 서명 키, 공식 tag, APT 게시 권한을 모두 자동으로 받지 않습니다. 필요한 최소 권한을 별도로 승인하고 사용 후 회수합니다.
- Security Triage 또는 행동강령 대응은 신고자 안전과 비밀 유지를 우선합니다. 이해관계가 있는 담당자는 사건 판단에서 빠집니다.
- 단일 담당자 부재로 안전한 검토가 불가능하면 릴리스·게시·공개를 미루며, 긴급한 보안 완화는 최소 권한과 사후 검토 기록을 사용합니다.

## 보안과 행동강령

- 취약점, credential 노출, 서명·배포 신뢰 경로 문제는 [SECURITY.md](SECURITY.md)에 따라 비공개로 triage합니다. 공개 Issue에 재현 절차나 비밀값을 올리지 않습니다.
- 행동강령 신고와 보호 조치는 [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)를 따릅니다. 보복을 금지하며, 필요한 정보만 제한적으로 공유합니다.
- 보안 수정의 공개, CVE 요청, 릴리스 시점은 사용자 보호를 우선해 Security Triage 담당자와 Maintainer가 결정합니다.

## 릴리스·문서 책임

- Release Manager는 tag, 소스 commit, 공식 release asset, APT archive 서명 및 provenance를 같은 신뢰 경로로 확인합니다. 개인 미러는 공식 승인 원천이 아니라 배포 미러로만 취급합니다.
- 문서 담당자는 구현하지 않은 기능을 완료된 것처럼 쓰지 않으며, GPU/NCCL 검증은 실제 증적 없이 성공으로 기록하지 않습니다.
- 변경 이력은 [CHANGELOG.md](dure/CHANGELOG.md)에 사용자 영향 중심으로 기록합니다. 패키지·증적·복구 절차는 각각의 운영 문서를 원본으로 합니다.
