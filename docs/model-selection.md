# 모델 선택 정책

> 상태: **부분 구현**. 버전이 있는 Qwen2.5 정적 카탈로그와 결정론적 선택기, 중앙 모델 레지스트리, 벤치마크 증적 저장·`ACTIVE` 승격 게이트, 정확한 다중 노드 증적을 소비하는 중앙 추천 자격 게이트, 폐쇄형 단일 GPU 벤치마크 실행, 추천 스냅샷 저장과 명시적 수락, 배포 세대별 적용·검증 상태와 명시적 직전 세대 롤백, `FULL_SNAPSHOT` 기반 `VLLM_RAY_PP_V1` 2·3노드 rank 결합은 구현되었습니다. 다중 노드 네트워크·NCCL 시험 자동 실행, 실제 GPU stage 배포와 24시간 복구 검증은 아직 구현되지 않았습니다.

## 목표

GPU 서버가 추가되거나 노드 프로필이 바뀌면, Dure는 최신 인벤토리를 사용해 **목표 작업 부하의 SLO를 만족하는 검증된 후보 중 품질이 가장 높은 모델**을 추천합니다.

추천은 자동 배포가 아닙니다.

```text
노드 승인 또는 조사 완료
        ↓
인벤토리 스냅샷과 정책 평가
        ↓
추천 생성 및 운영자 검토
        ↓
명시적 승인으로 배포 구성 생성
        ↓
명시적 적용으로만 호스트 변경
```

추천 생성은 검토 가능한 불변 스냅샷을 저장하고, 명시적 수락은 적용 전 배포 세대 한 건만 만듭니다. 모델 다운로드, 이미지 내려받기, 배포 작업 생성, 컨테이너 시작·중지는 운영자의 별도 명령이 필요합니다.

## 현재 상태와 범위

현재 계획기는 버전이 있는 Qwen2.5 AWQ 7B·14B·32B·72B 정적 카탈로그와 물리 노드당 GPU 한 장을 전제로 합니다. 선택기는 노드 UUID 정렬, 승인·온라인·프로필 신선도, GPU·디스크·런타임·네트워크 증적을 결정론적으로 평가하고 구조화된 탈락 사유를 반환합니다. 로컬 `dure plan --model auto`는 기존 호환성을 유지하면서 같은 프로필 집합에서 같은 모델과 노드 배정을 선택합니다.

선택기의 기본 정책은 네트워크/NCCL 증적이 없는 다중 노드 후보를 거부합니다. 다만 로컬 `dure plan --model auto`는 기존 3×24GB 계획 호환성을 위해 이 검사만 예외로 두며, 생성된 계획에 트래픽 제공 전 네트워크 대역폭과 RTT를 검증하라는 경고를 포함합니다. 중앙 추천에는 이 예외를 적용하지 않습니다.

중앙 제어면은 영속 모델 레지스트리와 `POST /v1/admin/deployment-recommendations` 추천 API를 제공합니다. 추천기는 저장된 중앙 노드 UUID·프로필과 `ACTIVE` 릴리스의 모든 배치 프로필을 평가하며, 같은 입력에는 같은 콘텐츠 해시 추천 ID·후보·탈락 사유를 반환합니다. 추천 결과와 지문 계산에 사용한 정규화 인벤토리는 불변 스냅샷으로 한 번만 저장됩니다. 이 DB 기록 외에 배포·작업·호스트 상태는 바꾸지 않습니다.

중앙 다중 노드 후보는 정확히 정렬된 노드 UUID 조합에 대해 등록된 최신 구조화 네트워크·NCCL 증적을 조회합니다. 증적은 같은 모델 릴리스, 배치 프로필, 아티팩트 리비전·매니페스트, 런타임 릴리스·OCI 이미지와 현재 인벤토리 지문에 결합돼야 하고 등록 후 24시간 이내여야 합니다. 같은 조합의 최신 결과가 실패했거나 더 늦게 준비·실행 중·실패한 벤치마크 실행이 있거나, 시각이 미래이거나 식별자·지문이 다르면 과거 통과 결과를 사용하지 않습니다. 통과 증적의 대역폭·RTT·패킷 손실·NCCL 값도 현재 배치 프로필 임계값을 다시 만족해야 합니다. 여러 증적의 노드를 섞어 새로운 조합을 만들지 않습니다.

별도의 vLLM 단계 아티팩트 레지스트리는 원본·런타임·토폴로지 계약이 고정된 rank별 결과와 실제 GPU export/load 증적을 관리합니다. 현재 추천기는 이 레지스트리를 입력으로 사용하지 않으며, `VALIDATED` 단계 variant가 생겨도 모델 준비 작업이나 배포를 자동 생성하지 않습니다. 0.3.18의 다중 노드 추천은 `VLLM_RAY_PP_V1`과 각 노드의 전체 `FULL_SNAPSHOT`을 선택합니다. 추천과 stage variant 선택, rank별 준비와 `sharded_state` 로더의 결합은 다음 PR 범위입니다.

3×24GB 노드의 72B 파이프라인은 기존 기준점입니다. 다중 노드 시험 자동 실행과 다중 GPU 서버·이기종 복제본을 종합적으로 선택하는 기능은 아직 제공하지 않습니다.

중앙 다중 노드 세대를 만들 때는 서버가 발급한 canonical 노드 UUID와 고유한 RFC1918 IPv4를 사용합니다. head는 rank 0이고 worker는 IPv4 문자열 오름차순이며, 각 노드에는 정상 GPU가 정확히 한 장 있어야 합니다. 런타임은 정확히 vLLM 0.9.0 V0 Ray, `TP=1`, `PP=2/3`으로 고정합니다. 이 조건을 만족하지 않는 후보는 다른 rank나 backend를 추측하지 않고 구조화된 탈락 사유로 닫습니다. 기존 로컬 계획 JSON과 legacy backend는 전용 필드 없이 기존 동작을 유지합니다.

초기 정책은 기존 topology를 안전하게 평가하는 데 집중합니다.

- 단일 GPU: 7B·14B·32B 계열
- 3×24GB의 검증된 사설망: 72B pipeline
- 추가 GPU: 현재 runtime이 지원하지 않는 DP/TP를 추측해 사용하지 않고 spare로 보고

TP, PP+TP, DP, MoE EP는 각각 실제 런타임과 벤치마크가 검증된 뒤 배치 프로필에 추가합니다.

## 선택 기준

`quality-within-SLO`(SLO 범위 내 최고 품질) 정책은 숨은 가중합 대신 다음 순서로 후보를 평가합니다.

1. **실행 가능성**: 승인·온라인·신선한 프로필, 정상 GPU, Docker/NVIDIA 런타임, 디스크, 드라이버, 연산 능력, 이미지와 모델 아티팩트 조건을 모두 만족해야 합니다.
2. **네트워크 조건**: 중앙 추천의 다중 노드 후보는 정확한 노드 UUID 조합의 24시간 이내 최신 `PASSED` 증적으로 RTT, 대역폭, 패킷 손실과 NCCL 검증을 통과해야 합니다. 다른 조합의 증적이나 과거 통과 결과를 합쳐 사용할 수 없습니다.
3. **메모리 여유**: 가중치 조각, KV 캐시, 활성화·작업 공간, 안전 여유를 모두 고려해 VRAM 여유를 남겨야 합니다.
4. **품질**: 해당 작업 부하의 내부 평가 점수가 높은 후보를 우선합니다.
5. **서비스 수준**: TTFT, TPOT, 종단 간 지연, 유효 처리량, 성공률이 정책 SLO를 만족해야 합니다.
6. **동일 품질 후보**: 완전한 로컬 아티팩트 캐시, 적은 노드 수, 높은 벤치마크 여유, 정렬된 노드 UUID 순으로 결정합니다.

입력 프로필의 순서가 결과에 영향을 주어서는 안 됩니다. 결정에는 카탈로그 버전, 정책 버전, 정렬된 노드 UUID, 인벤토리 지문을 기록합니다.

## 모델 레지스트리

중앙 모델 레지스트리는 모델 이름만 저장하지 않습니다.

| 항목 | 목적 |
|---|---|
| 모델 아티팩트 | 저장소, 변경 불가능한 리비전, 매니페스트 다이제스트, 양자화, 크기, 라이선스 |
| 런타임 릴리스 | vLLM/CUDA 버전, OCI 이미지 다이제스트, GPU 세대 호환성 |
| 모델 릴리스 | 아티팩트와 런타임의 검증된 조합, `DRAFT → VALIDATED → ACTIVE` 상태 |
| 배치 프로필 | 지원 토폴로지, 최소 VRAM·디스크, TP/PP/DP/EP, 네트워크 조건 |
| 벤치마크 실행 | 품질, 성능, 안정성, 하드웨어·드라이버·정책 증적 |

레지스트리는 현재 `DRAFT`, `VALIDATED`, `ACTIVE`, `DEPRECATED`, `REVOKED` 상태와 관리자 API를 제공합니다. 브랜치나 태그가 아닌 변경 불가능한 커밋 리비전을 사용하며, 런타임 이미지 역시 OCI 다이제스트로 고정합니다. 배치 프로필은 `DRAFT` 상태에서만 추가할 수 있습니다.

`VALIDATED` 릴리스를 `ACTIVE`로 승격하려면 모든 배치 프로필의 최신 증적이 통과해야 합니다. 증적은 릴리스·배치 프로필, 정렬된 중앙 노드 UUID와 현재 프로필 지문에 결합되며, 아티팩트 리비전·매니페스트와 OCI 런타임 이미지가 레지스트리 값과 정확히 일치해야 합니다. 최신 결과가 실패했거나 프로필이 바뀌었다면 과거 통과 결과로 승격할 수 없습니다.

승격된 릴리스라도 중앙 다중 노드 추천 시점에는 후보의 정확한 노드 조합에 맞는 최신 증적을 다시 검사합니다. 추천 자격의 24시간 TTL은 장기 상태인 `ACTIVE` 승격을 자동 취소하는 규칙이 아니라, 오래된 네트워크 측정을 새 배포 후보에 재사용하지 않기 위한 별도 읽기 게이트입니다.

승격에 사용한 증적 집합은 릴리스에 고정되어 반복 승격 요청이 상태나 감사 기록을 중복 변경하지 않습니다. 증적 등록과 승격만으로 에이전트 작업, 다운로드, Docker 실행 또는 배포 구성이 생기지 않습니다.

중앙 읽기 전용 추천에는 `ACTIVE` 상태의 릴리스만 사용할 수 있습니다. 추천 API는 이 릴리스들을 현재 인벤토리와 결정론적으로 비교하며, 관리자 상태 전이와 추천 조회는 자동 배포를 유발하지 않습니다.

레지스트리는 임의 Docker 인자, 환경 변수, 호스트 마운트 경로를 저장하거나 에이전트에 전달하지 않습니다. 런타임 옵션은 형식화된 허용 목록으로 제한합니다.

## 후보 모델과 양자화

기존 Qwen2.5 AWQ 모델은 안정적인 대체 후보로 유지합니다. 신규 후보는 별도 벤치마크와 런타임 적격성 검사를 거쳐 추가합니다. 후보의 품질은 파라미터 수만으로 판단하지 않습니다.

GPU 세대별 양자화 호환성도 배치 조건입니다. 예를 들어 하나의 FP8 아티팩트가 모든 NVIDIA GPU에서 같은 방식으로 실행된다고 가정하지 않으며, Ampere·Ada·Hopper 등 실제 하드웨어와 vLLM 런타임 조합을 각각 검증합니다.

## 현재 자격 검증 API

관리자는 다음 API로 외부에서 측정한 구조화된 증적을 등록하고 승격을 요청할 수 있습니다.

- `POST /v1/admin/benchmark-context`
- `POST /v1/admin/benchmark-runs/prepare`
- `POST /v1/admin/benchmark-runs/{request_id}/apply`
- `GET /v1/admin/benchmark-runs/{request_id}`
- `POST /v1/admin/benchmark-evidence`
- `GET /v1/admin/benchmark-evidence`
- `POST /v1/admin/model-releases/{id}/promote`

context API는 선택한 릴리스·배치 프로필·노드 UUID를 확인하고 증적에 사용할 현재 프로필 지문과 고정 식별자를 반환합니다. 중앙 제어면은 수동 증적을 계속 받을 수 있고, 준비와 명시적 적용을 분리한 폐쇄형 `BENCHMARK` 작업으로 승인된 단일 GPU 노드의 고정 작업 부하도 실행할 수 있습니다. 자동 경로는 펼쳐진 로컬 Dure 모델 캐시와 다이제스트 고정 이미지만 사용하며 다운로드·이미지 내려받기·배포 생성을 수행하지 않습니다. 전용 관리자 CLI, 다중 노드 네트워크·NCCL 실행과 전체 작업 부하 매트릭스는 후속 범위입니다. 전체 지표와 제한 사항은 [벤치마크 문서](benchmarking.md)를 참고합니다.

## 추천 조회와 명시적 수락

다음 명령을 사용할 수 있습니다. `recommend`는 저장 프로필을 평가할 뿐 프로필을 갱신하지 않으므로, 필요하면 별도 `probe` 작업을 먼저 실행하고 완료를 확인합니다. 로컬 `dure plan --model auto`와 명시적 `--model` 경로도 계속 사용할 수 있습니다.

```bash
dure admin probe --nodes <node-id> <node-id>
dure admin deployment recommend --all-online
dure admin deployment recommend --nodes <node-id> <node-id> --objective quality-first
dure admin recommendation show <recommendation-id>
dure admin recommendation accept <recommendation-id>
dure admin recommendation accept <recommendation-id> \
  --previous-generation <deployment-id>
# 실제 호스트 변경은 여전히 별도 명령입니다.
dure admin apply <deployment-id> --nodes <node-id> ...
```

추천 응답에는 콘텐츠 해시 ID, 카탈로그·정책 버전, 정렬된 중앙 노드 UUID, 인벤토리 지문, 선택 후보와 모든 탈락 사유가 들어 있습니다. 다중 노드 후보가 선택되면 정확한 조합을 적격으로 만든 증적 ID와 다이제스트도 후보 스냅샷에 결합됩니다. 서버는 이 응답과 지문에 사용한 승인·연결·프로필 신선도·정규화 프로필을 함께 저장하며, 같은 콘텐츠 ID의 반복 추천은 행을 중복 생성하지 않습니다. `show`는 이 스냅샷과 이미 수락된 세대가 있는지를 조회합니다.

`accept`는 저장된 선택 모드로 현재 추천을 다시 계산합니다. 콘텐츠 ID, 카탈로그·정책 버전, 인벤토리 지문과 정규화 스냅샷, 선택 릴리스·배치·노드가 하나라도 달라졌거나 실행 가능한 선택 후보가 없으면 `409`로 거부합니다. 성공하면 OCI 다이제스트·모델 리비전·중앙 노드 UUID를 고정한 `CREATED` 배포 세대만 저장합니다. 중앙 세대의 모델 경로는 probe가 조사할 수 있는 단일 디렉터리인 `/var/lib/dure/models/<model-id>--<revision>` 형식으로 고정해 같은 모델 ID의 다른 리비전을 혼동하지 않습니다. `accept_model_download`와 `pull_image`는 항상 거짓이며 작업·벤치마크 실행·다운로드·Docker 변경은 만들지 않습니다.

다중 노드 선택이면 생성된 plan은 `execution_backend=VLLM_RAY_PP_V1`, `runtime_vllm_version=0.9.0`, `model_cache_kind=FULL_SNAPSHOT`과 전체 ordered binding을 포함합니다. 수락 뒤에도 일부 노드만 대상으로 비중지 작업을 만들 수 없고 모든 대상 Agent가 0.3.18 이상이어야 합니다. UUID를 hostname으로 보정하는 legacy 호환 경로는 이 엄격한 backend에 적용하지 않습니다.

첫 수락은 generation 1의 새 계보를 만듭니다. `--previous-generation`을 지정하면 그 세대의 계보를 이어 generation을 하나 증가시키며, 지정한 세대가 해당 계보의 최신 세대가 아니거나 `ROLLED_BACK` 상태이거나 활성 operation·변경 task가 있으면 분기 생성을 거부합니다. 같은 추천과 같은 이전 세대를 반복 수락하면 기존 세대를 반환하고 감사 이벤트를 중복 생성하지 않습니다. 롤백으로 `ROLLED_BACK`이 된 최신 세대는 과거 계보를 이어 갈 이전 세대로 사용할 수 없으므로, 롤백 뒤 새 추천을 수락할 때는 이전 세대를 생략하고 새 계보를 시작해야 합니다. 현재 계획 형식이 tensor rank를 표현하지 못하므로 단일 GPU와 기존 pipeline-parallel 표현 범위 밖의 배치는 수락 단계에서 실패 안전 방식으로 거부합니다.

## 세대 상태 조회와 롤백

명시적 `apply`와 `verify`는 세대별 operation, 노드 상태와 task 시도 번호를 기록합니다. operation은 `PREPARED`, `QUEUED`, `RUNNING`, `SUCCEEDED`, `PARTIAL_FAILED`, `FAILED`를 사용하고 노드별 단계는 `PENDING`, `QUEUED`, `RUNNING`, `SUCCEEDED`, `FAILED`, `CANCELED`를 사용합니다. 현재 operation 단계와 시도 번호에 맞지 않는 늦은 task 보고는 상태를 변경하지 않습니다.

```bash
dure admin deployment show <deployment-id>
dure admin deployment generations <deployment-id>

# 준비만 하며 task는 만들지 않습니다.
dure admin deployment rollback <latest-deployment-id> \
  --nodes <node-a> <node-b> --serve

# 같은 입력에 --apply를 추가해야 실제 롤백을 시작합니다.
dure admin deployment rollback <latest-deployment-id> \
  --nodes <node-a> <node-b> --serve --apply
```

관리자 API는 `GET /v1/admin/deployments/{deployment_id}`, `GET /v1/admin/deployments/{deployment_id}/generations`, `POST /v1/admin/deployments/{source_id}/rollback`을 제공합니다. 롤백 본문은 1개 이상 64개 이하의 정규 UUID `node_ids`와 기본값이 `false`인 엄격한 불리언 `apply`, `serve`만 받습니다. 대상 세대, 계획, 다운로드나 pull 입력은 받지 않으며 서버가 저장된 계보에서 대상을 결정합니다.

`verified_at`은 계획의 전체 배정 노드를 대상으로 한 검증이 모두 성공하고 legacy Agent는 0.3.12 이상, `VLLM_RAY_PP_V1` Agent는 0.3.18 이상이며 엄격한 rank·API 검증까지 통과할 때만 기록됩니다. 일부 노드나 전체 배정 집합을 충족하지 않는 Ray head 전용 API 검증은 롤백 증거가 아닙니다. 롤백은 계보의 최신 소스에서 그 소스가 직접 가리키는 상태가 `VERIFIED`이고 `verified_at`을 보유한 직전 세대로만 가능하고, 두 세대의 전체 배정과 토폴로지가 정확히 같아야 합니다. 요청 노드 집합도 전체 배정과 정확히 같아야 하며 모든 노드는 승인됨·온라인이고 같은 backend별 최소 버전을 충족해야 합니다. 양쪽 이미지가 OCI 다이제스트로 고정되지 않았으면 거부합니다.

기본 요청은 `PREPARED` operation만 저장합니다. `apply=true` 뒤에만 `STOP_SOURCE → START_TARGET(serve=false) → VERIFY_TARGET` 순서로 모든 노드를 처리하며, `serve=true`이면 Ray head 한 대에서 `START_API → VERIFY_API`를 이어서 수행합니다. `VLLM_RAY_PP_V1` rollback은 actor 증적 없는 세대를 검증 완료로 만들지 않도록 `serve=true`가 필수입니다. 한 단계의 모든 노드가 성공해야 다음 단계로 넘어갑니다. 실패 뒤 같은 입력에 `apply=true`를 다시 보내면 현재 단계의 실패 노드만 증가한 시도 번호로 재시도합니다.

일반 apply에서 `serve=true`를 사용할 때도 전체 배정 노드가 필요합니다. 모든 노드의 Ray 준비가 끝난 뒤에만 head 전용 API 시작·검증 단계가 생성됩니다. 전체 verify 결과는 단계별 필수 검사 이름이 빠지거나 중복되면 증거로 인정하지 않으며, 같은 노드를 공유하는 다른 계보의 활성 operation도 거부합니다.

롤백은 모델 다운로드나 이미지 내려받기를 하지 않습니다. 대상 캐시와 다이제스트 이미지를 미리 준비해야 합니다. 같은 GPU에서 소스 컨테이너를 중지하고 대상 세대를 다시 만드는 방식이므로 중단이 발생할 수 있으며 블루·그린 배포가 아닙니다. 이 세대 검증은 네트워크·NCCL 시험이나 24시간 복구 검증을 대신하지 않습니다.

## 안전 경계

- 대기 중, 폐기됨, 오프라인, 오래된 노드는 선택하지 않으며 명시적으로 요청했을 때 구조화된 탈락 사유를 반환합니다.
- 명시적 `--model`은 운영자의 복구 및 예외 처리 수단으로 유지합니다.
- Codex 기반 `dure admin diagnose`는 참고용이며 모델 선택의 결정적 입력으로 사용하지 않습니다.
- 프로필이 바뀌었다고 자동으로 기존 서비스를 교체하지 않습니다. 최소 개선폭, 대기 시간, 안정성 조건을 충족한 뒤에만 운영자가 승격을 검토합니다.
- 중앙 추천의 다중 노드 모델은 정확한 정렬 노드 UUID 조합·현재 인벤토리·릴리스·배치·런타임에 결합된 24시간 이내 최신 통과 네트워크/NCCL 증적이 없으면 차단 사전 조건을 반환합니다. 최신 실패·진행 중 실행, 미래·만료·불일치 증적도 실패 안전 방식으로 차단합니다. 로컬 `dure plan --model auto`의 기존 호환 예외는 계획 경고를 남깁니다.
- `ACTIVE` 승격은 각 배치 프로필의 최신 증적만 사용하며, 실패한 최신 결과를 과거 통과 결과로 대체하지 않습니다.
- 증적 등록과 승격은 실행 중인 배포를 자동 교체하거나 강등하지 않습니다.
- 추천 저장은 불변 검토 기록만 추가하고, 추천 수락은 적용 전 배포 세대만 추가합니다. 어느 쪽도 호스트 상태를 바꾸지 않습니다.
- 오래된 추천, 변경된 인벤토리·카탈로그·선택 결과는 수락할 수 없습니다.
- 이전 세대를 지정한 수락은 해당 계보의 최신 세대만 이어 갈 수 있습니다.
- 모델 캐시 경로는 모델 ID만이 아니라 리비전 또는 매니페스트 다이제스트를 포함해야 합니다.
- 롤백 준비는 task나 호스트 변경을 만들지 않고, `apply=true`가 있어야만 적용을 시작합니다.
- 롤백 대상은 클라이언트가 고르지 않으며 서버가 최신 소스의 검증된 직전 세대로 제한합니다.
- 새 컨테이너는 정확한 배포·세대·노드 레이블을 사용하고, 불일치하는 컨테이너는 중지·제거·재사용하지 않습니다.
- `VLLM_RAY_PP_V1` 컨테이너는 backend·pipeline rank·runtime rank·component 레이블도 모두 일치해야 하며 GCS 6379, worker 20000-21000, loopback API 8000만 사용합니다.
- `pipeline-rank-contract`는 vLLM 0.9.0 소스 규칙과 Ray topology의 간접 증적이지 Ray가 내부 rank를 직접 보고한 값이 아닙니다. 불확실한 topology는 실패로 닫습니다.
- preflight 실패는 새 컨테이너를 시작하지 않습니다. 실행 중 실패는 자동 driver 변경이나 자동 failover로 숨기지 않고, 실패 노드 격리 후 명시적 재시도 또는 검증된 직전 세대 롤백으로 처리합니다.
- rank별 `STAGE` 선택·준비·로더 결합과 probe 기반 캐시 상태 투영은 아직 계획입니다. 현재 추천은 `FULL_SNAPSHOT`만 선택합니다.

## 완료 기준

- **구현됨:** 동일 인벤토리는 항상 동일한 추천과 탈락 사유를 만듭니다.
- **구현됨:** 선택기의 기본 정책에서 3×22GB, 대기 중 노드, 오래된 프로필, 네트워크 증적 누락은 72B 후보를 통과하지 못합니다. 로컬 계획기는 위 호환 예외를 유지합니다.
- **구현됨:** 중앙 다중 노드 추천은 정확한 노드 조합의 최신 통과 증적만 사용하고, 24시간 TTL·현재 지문·런타임 식별자·배치 네트워크 임계값을 다시 검사하며 서로 다른 조합의 노드를 섞지 않습니다.
- **구현됨:** GPU를 추가해도 실행 가능한 후보 중 품질 순위가 가장 높은 모델을 결정론적으로 선택합니다.
- **구현됨:** 추천 생성만으로 배포 구성, 작업, 다운로드, Docker 실행이 발생하지 않습니다.
- **구현됨:** 추천을 불변 스냅샷으로 조회하고, 유효성을 재검사한 명시적 수락으로만 적용 전 배포 세대를 만듭니다.
- **구현됨:** 같은 수락은 멱등적이며, 변경된 인벤토리와 최신이 아닌 이전 세대는 거부합니다.
- **구현됨:** 적용·검증 operation과 노드·시도 상태를 조회하고 전체 노드 검증만 롤백 증거로 기록합니다.
- **구현됨:** 준비와 명시적 적용을 분리하고 직전 검증 세대·동일 토폴로지·전체 안전 노드만 롤백합니다.
- **구현됨:** 기존 계획 JSON과 명시적 `--model` 동작이 유지됩니다.
- **구현됨:** 중앙 2·3노드 추천 세대가 서버 UUID·고유 RFC1918 주소와 고정 vLLM 0.9.0 Ray 계약에 결정론적으로 결합됩니다.
- 신규 모델은 벤치마크·보안·아티팩트 검증을 통과한 뒤에만 `ACTIVE`가 됩니다.
- 다중 노드 네트워크·NCCL 시험 자동 실행, 실제 GPU stage 배포와 24시간 복구 수용 검사는 별도 실제 환경에서 완료해야 합니다.
