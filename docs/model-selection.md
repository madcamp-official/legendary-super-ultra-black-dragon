# 모델 선택 정책

> 상태: **부분 구현**. 버전이 있는 Qwen2.5 정적 카탈로그와 결정론적 선택기, 중앙 모델 레지스트리, 저장 인벤토리 기반 읽기 전용 추천, 폐쇄형 단일 GPU 벤치마크 실행과 증적 저장·`ACTIVE` 승격 게이트는 구현되었습니다. 추천 승인·배포 세대 전환과 추천기의 네트워크·NCCL 증적 연계는 아직 구현되지 않았습니다.

## 목표

GPU 서버가 추가되거나 노드 프로필이 바뀌면, Dure는 최신 인벤토리를 사용해 **목표 작업 부하의 SLO를 만족하는 검증된 후보 중 품질이 가장 높은 모델**을 추천합니다.

추천은 자동 배포가 아닙니다.

```text
노드 승인 또는 조사 완료
        ↓
인벤토리 스냅샷과 정책 평가
        ↓
추천 생성 및 운영자 검토
        ↓
명시적 승인으로 배포 구성 생성
        ↓
명시적 적용으로만 호스트 변경
```

초기에는 추천 생성만 자동화합니다. 모델 다운로드, 이미지 내려받기, 배포 작업 생성, 컨테이너 시작·중지는 운영자의 별도 명령이 필요합니다.

## 현재 상태와 범위

현재 계획기는 버전이 있는 Qwen2.5 AWQ 7B·14B·32B·72B 정적 카탈로그와 물리 노드당 GPU 한 장을 전제로 합니다. 선택기는 노드 UUID 정렬, 승인·온라인·프로필 신선도, GPU·디스크·런타임·네트워크 증적을 결정론적으로 평가하고 구조화된 탈락 사유를 반환합니다. 로컬 `dure plan --model auto`는 기존 호환성을 유지하면서 같은 프로필 집합에서 같은 모델과 노드 배정을 선택합니다.

선택기의 기본 정책은 네트워크/NCCL 증적이 없는 다중 노드 후보를 거부합니다. 다만 로컬 `dure plan --model auto`는 기존 3×24GB 계획 호환성을 위해 이 검사만 예외로 두며, 생성된 계획에 트래픽 제공 전 네트워크 대역폭과 RTT를 검증하라는 경고를 포함합니다. 중앙 추천에는 이 예외를 적용하지 않습니다.

중앙 제어면은 영속 모델 레지스트리와 `POST /v1/admin/deployment-recommendations` 읽기 전용 추천 API를 제공합니다. 추천기는 저장된 중앙 노드 UUID·프로필과 `ACTIVE` 릴리스의 모든 배치 프로필을 평가하며, 같은 입력에는 같은 콘텐츠 해시 추천 ID·후보·탈락 사유를 반환합니다. 3×24GB 노드의 72B 파이프라인은 기존 기준점입니다. 구조화된 네트워크·NCCL 증적 저장은 구현되었지만 추천기가 이를 노드 조합의 적격성 입력으로 조회하는 연계는 아직 없어 중앙 다중 노드 후보는 현재 모두 실패 안전 방식으로 거부됩니다. 다중 GPU 서버·이기종 복제본을 종합적으로 선택하는 기능도 아직 제공하지 않습니다.

초기 정책은 기존 topology를 안전하게 평가하는 데 집중합니다.

- 단일 GPU: 7B·14B·32B 계열
- 3×24GB의 검증된 사설망: 72B pipeline
- 추가 GPU: 현재 runtime이 지원하지 않는 DP/TP를 추측해 사용하지 않고 spare로 보고

TP, PP+TP, DP, MoE EP는 각각 실제 런타임과 벤치마크가 검증된 뒤 배치 프로필에 추가합니다.

## 선택 기준

`quality-within-SLO`(SLO 범위 내 최고 품질) 정책은 숨은 가중합 대신 다음 순서로 후보를 평가합니다.

1. **실행 가능성**: 승인·온라인·신선한 프로필, 정상 GPU, Docker/NVIDIA 런타임, 디스크, 드라이버, 연산 능력, 이미지와 모델 아티팩트 조건을 모두 만족해야 합니다.
2. **네트워크 조건**: 중앙 추천과 선택기의 기본 정책에서 다중 노드 후보는 RTT, 대역폭, 패킷 손실, 방화벽, NCCL 검증을 통과해야 합니다.
3. **메모리 여유**: 가중치 조각, KV 캐시, 활성화·작업 공간, 안전 여유를 모두 고려해 VRAM 여유를 남겨야 합니다.
4. **품질**: 해당 작업 부하의 내부 평가 점수가 높은 후보를 우선합니다.
5. **서비스 수준**: TTFT, TPOT, 종단 간 지연, 유효 처리량, 성공률이 정책 SLO를 만족해야 합니다.
6. **동일 품질 후보**: 완전한 로컬 아티팩트 캐시, 적은 노드 수, 높은 벤치마크 여유, 정렬된 노드 UUID 순으로 결정합니다.

입력 프로필의 순서가 결과에 영향을 주어서는 안 됩니다. 결정에는 카탈로그 버전, 정책 버전, 정렬된 노드 UUID, 인벤토리 지문을 기록합니다.

## 모델 레지스트리

중앙 모델 레지스트리는 모델 이름만 저장하지 않습니다.

| 항목 | 목적 |
|---|---|
| 모델 아티팩트 | 저장소, 변경 불가능한 리비전, 매니페스트 다이제스트, 양자화, 크기, 라이선스 |
| 런타임 릴리스 | vLLM/CUDA 버전, OCI 이미지 다이제스트, GPU 세대 호환성 |
| 모델 릴리스 | 아티팩트와 런타임의 검증된 조합, `DRAFT → VALIDATED → ACTIVE` 상태 |
| 배치 프로필 | 지원 토폴로지, 최소 VRAM·디스크, TP/PP/DP/EP, 네트워크 조건 |
| 벤치마크 실행 | 품질, 성능, 안정성, 하드웨어·드라이버·정책 증적 |

레지스트리는 현재 `DRAFT`, `VALIDATED`, `ACTIVE`, `DEPRECATED`, `REVOKED` 상태와 관리자 API를 제공합니다. 브랜치나 태그가 아닌 변경 불가능한 커밋 리비전을 사용하며, 런타임 이미지 역시 OCI 다이제스트로 고정합니다. 배치 프로필은 `DRAFT` 상태에서만 추가할 수 있습니다.

`VALIDATED` 릴리스를 `ACTIVE`로 승격하려면 모든 배치 프로필의 최신 증적이 통과해야 합니다. 증적은 릴리스·배치 프로필, 정렬된 중앙 노드 UUID와 현재 프로필 지문에 결합되며, 아티팩트 리비전·매니페스트와 OCI 런타임 이미지가 레지스트리 값과 정확히 일치해야 합니다. 최신 결과가 실패했거나 프로필이 바뀌었다면 과거 통과 결과로 승격할 수 없습니다.

승격에 사용한 증적 집합은 릴리스에 고정되어 반복 승격 요청이 상태나 감사 기록을 중복 변경하지 않습니다. 증적 등록과 승격만으로 에이전트 작업, 다운로드, Docker 실행 또는 배포 구성이 생기지 않습니다.

중앙 읽기 전용 추천에는 `ACTIVE` 상태의 릴리스만 사용할 수 있습니다. 추천 API는 이 릴리스들을 현재 인벤토리와 결정론적으로 비교하며, 관리자 상태 전이와 추천 조회는 자동 배포를 유발하지 않습니다.

레지스트리는 임의 Docker 인자, 환경 변수, 호스트 마운트 경로를 저장하거나 에이전트에 전달하지 않습니다. 런타임 옵션은 형식화된 허용 목록으로 제한합니다.

## 후보 모델과 양자화

기존 Qwen2.5 AWQ 모델은 안정적인 대체 후보로 유지합니다. 신규 후보는 별도 벤치마크와 런타임 적격성 검사를 거쳐 추가합니다. 후보의 품질은 파라미터 수만으로 판단하지 않습니다.

GPU 세대별 양자화 호환성도 배치 조건입니다. 예를 들어 하나의 FP8 아티팩트가 모든 NVIDIA GPU에서 같은 방식으로 실행된다고 가정하지 않으며, Ampere·Ada·Hopper 등 실제 하드웨어와 vLLM 런타임 조합을 각각 검증합니다.

## 현재 자격 검증 API

관리자는 다음 API로 외부에서 측정한 구조화된 증적을 등록하고 승격을 요청할 수 있습니다.

- `POST /v1/admin/benchmark-context`
- `POST /v1/admin/benchmark-runs/prepare`
- `POST /v1/admin/benchmark-runs/{request_id}/apply`
- `GET /v1/admin/benchmark-runs/{request_id}`
- `POST /v1/admin/benchmark-evidence`
- `GET /v1/admin/benchmark-evidence`
- `POST /v1/admin/model-releases/{id}/promote`

context API는 선택한 릴리스·배치 프로필·노드 UUID를 확인하고 증적에 사용할 현재 프로필 지문과 고정 식별자를 반환합니다. 중앙 제어면은 수동 증적을 계속 받을 수 있고, 준비와 명시적 적용을 분리한 폐쇄형 `BENCHMARK` 작업으로 승인된 단일 GPU 노드의 고정 작업 부하도 실행할 수 있습니다. 자동 경로는 펼쳐진 로컬 Dure 모델 캐시와 다이제스트 고정 이미지만 사용하며 다운로드·이미지 내려받기·배포 생성을 수행하지 않습니다. 전용 관리자 CLI, 다중 노드 네트워크·NCCL 실행과 전체 작업 부하 매트릭스는 후속 범위입니다. 전체 지표와 제한 사항은 [벤치마크 문서](benchmarking.md)를 참고합니다.

## 현재 추천 API와 계획된 승인 흐름

다음의 `probe`와 `deployment recommend`는 현재 사용할 수 있습니다. `recommend`는 저장 프로필만 읽으므로 프로필 갱신이 필요하면 별도 `probe` 작업을 먼저 실행하고 완료를 확인합니다. `recommendation show`, `accept`, 세대별 적용은 후속 구현 범위입니다. 로컬 `dure plan --model auto`도 계속 사용할 수 있습니다.

```bash
dure admin probe --nodes <node-id> <node-id>
dure admin deployment recommend --all-online
dure admin deployment recommend --nodes <node-id> <node-id> --objective quality-first
# 아래 명령은 아직 계획 단계입니다.
dure admin recommendation show <recommendation-id>
dure admin recommendation accept <recommendation-id>
dure admin apply <deployment-id> --nodes <node-id> ...
```

추천 응답에는 콘텐츠 해시 ID, 카탈로그·정책 버전, 정렬된 중앙 노드 UUID, 인벤토리 지문, 선택 후보와 모든 탈락 사유가 들어 있습니다. 결과는 아직 영속 저장하지 않으므로 같은 추천 ID를 나중에 조회할 수 없습니다. 계획된 `accept`는 인벤토리 지문이 여전히 유효한지 확인한 뒤 배포 구성만 만들며, 명시적 `apply` 전에는 에이전트 작업이나 Docker 변경이 생기지 않아야 합니다.

## 안전 경계

- 대기 중, 폐기됨, 오프라인, 오래된 노드는 선택하지 않으며 명시적으로 요청했을 때 구조화된 탈락 사유를 반환합니다.
- 명시적 `--model`은 운영자의 복구 및 예외 처리 수단으로 유지합니다.
- Codex 기반 `dure admin diagnose`는 참고용이며 모델 선택의 결정적 입력으로 사용하지 않습니다.
- 프로필이 바뀌었다고 자동으로 기존 서비스를 교체하지 않습니다. 최소 개선폭, 대기 시간, 안정성 조건을 충족한 뒤에만 운영자가 승격을 검토합니다.
- 중앙 추천과 선택기의 기본 정책에서 다중 노드 모델은 네트워크/NCCL 증적이 없으면 차단 사전 조건을 반환합니다. 로컬 `dure plan --model auto`의 기존 호환 예외는 계획 경고를 남깁니다.
- `ACTIVE` 승격은 각 배치 프로필의 최신 증적만 사용하며, 실패한 최신 결과를 과거 통과 결과로 대체하지 않습니다.
- 증적 등록과 승격은 실행 중인 배포를 자동 교체하거나 강등하지 않습니다.
- 모델 캐시 경로는 모델 ID만이 아니라 리비전 또는 매니페스트 다이제스트를 포함해야 합니다.

## 완료 기준

- **구현됨:** 동일 인벤토리는 항상 동일한 추천과 탈락 사유를 만듭니다.
- **구현됨:** 선택기의 기본 정책에서 3×22GB, 대기 중 노드, 오래된 프로필, 네트워크 증적 누락은 72B 후보를 통과하지 못합니다. 로컬 계획기는 위 호환 예외를 유지합니다.
- **구현됨:** GPU를 추가해도 실행 가능한 후보 중 품질 순위가 가장 높은 모델을 결정론적으로 선택합니다.
- **구현됨:** 추천 생성만으로 배포 구성, 작업, 다운로드, Docker 실행이 발생하지 않습니다.
- **구현됨:** 기존 계획 JSON과 명시적 `--model` 동작이 유지됩니다.
- 신규 모델은 벤치마크·보안·아티팩트 검증을 통과한 뒤에만 `ACTIVE`가 됩니다.
