# 벤치마크 및 모델 자격 검증

> 상태: **부분 구현**. 중앙 제어면의 벤치마크 증적 저장, 고정 식별자 검증, SLO 판정, `ACTIVE` 승격 게이트와 저장 인벤토리 기반 읽기 전용 추천은 구현되었습니다. GPU 노드에서 벤치마크를 직접 실행하는 Agent 작업, 전체 작업 부하 매트릭스 실행, 다중 노드 24시간 복구 검증은 후속 범위입니다.

## 현재 제공 범위

중앙 제어면은 관리자 인증을 거친 구조화된 증적만 저장합니다. 증적 등록 자체는 Agent 작업, 모델 다운로드, 이미지 내려받기, Docker 실행 또는 배포 변경을 만들지 않습니다.

- 고정 suite `dure-serving-slo-v1`과 정책 `benchmark-gate-v1`만 받습니다.
- 모델 릴리스, 배치 프로필, 정렬된 중앙 노드 UUID, 현재 노드 프로필 지문을 하나의 증적에 결합합니다.
- 모델 아티팩트 리비전·매니페스트 다이제스트와 OCI 런타임 이미지가 레지스트리 값과 정확히 같아야 합니다.
- Dure 커밋, 입력·출력 토큰 수, 동시성, 예열·측정 요청 수와 측정 시간을 기록합니다.
- 품질, 지연 시간, 처리량, 성공률, VRAM 여유, OOM·충돌·재시작, 네트워크와 NCCL 결과를 형식화된 필드로 저장합니다.
- 서버가 판정한 `PASSED` 또는 `FAILED` 상태와 고정된 실패 코드를 저장합니다. SLO 미달 결과도 버리지 않고 `FAILED` 증적으로 남깁니다.
- 동일 내용은 증적 다이제스트로 중복 제거하며, 배치 프로필별 등록 순번으로 최신 결과를 판정합니다.

증적 API는 프롬프트, 자격 증명, 모델 접근 토큰, 로그 본문, 임의 명령, Docker 인자, 환경 변수, 마운트, 호스트 경로 또는 자유 형식 metadata 필드를 받지 않습니다.

## 증적과 릴리스 상태

배치 프로필은 `DRAFT` 릴리스에만 추가할 수 있습니다. 프로필 구성을 마친 뒤 릴리스를 `VALIDATED`로 전환하고 외부에서 측정한 증적을 등록합니다.

`VALIDATED → ACTIVE` 승격에는 다음 조건이 모두 필요합니다.

1. 릴리스의 모든 배치 프로필에 증적이 있어야 합니다.
2. 각 배치 프로필에서 등록 순번이 가장 큰 최신 증적이 `PASSED`여야 합니다. 최신 결과가 실패했다면 과거 통과 결과로 우회하지 않습니다.
3. 증적의 아티팩트 리비전, 모델 매니페스트와 런타임 이미지가 현재 레지스트리 값과 같아야 합니다.
4. 증적에 결합된 노드가 계속 승인 상태이고, 현재 프로필 지문이 측정 당시 지문과 같아야 합니다.

승격에 사용한 증적 ID 집합과 그 다이제스트는 모델 릴리스에 고정합니다. 같은 릴리스의 승격을 반복 요청하면 새 상태 전이나 감사 이벤트를 만들지 않고 최초 증적 집합을 반환합니다. `ACTIVE` 이후 새 실패 증적이 등록돼도 자동 강등하거나 실행 중인 배포를 변경하지 않습니다. 재검증 실패에 따른 `DEPRECATED` 또는 `REVOKED` 전환은 운영자가 별도로 판단해야 합니다.

## 현재 승격 판정

현재 정책은 다음 조건을 평가합니다.

- 예열 요청 20회 이상
- 측정 요청 200회 이상 또는 측정 시간 15분 이상
- OOM, 프로세스 충돌과 예기치 않은 런타임 재시작 없음
- 품질 점수 0.80 이상
- 배치 프로필의 p95 TTFT, TPOT, 종단 간 지연 상한 만족
- 배치 프로필의 최소 처리량, 요청 성공률과 VRAM 여유 만족
- 다중 노드 배치의 최소 대역폭, 최대 RTT와 패킷 손실 상한 만족
- NCCL이 필요한 배치의 집합 연산 증적 존재 및 성공

누락과 실패는 OOM, 충돌, 예열·측정량, 품질, TTFT·TPOT·종단 간 지연, 처리량, 성공률, VRAM, 네트워크, NCCL 범주별 코드로 구분합니다. 수치 필드는 NaN이나 무한대를 허용하지 않습니다.

## 관리자 API

- `POST /v1/admin/benchmark-context`: 릴리스·배치 프로필·노드 UUID를 검증하고 현재 인벤토리 지문과 고정 식별자 조회
- `POST /v1/admin/benchmark-evidence`: 구조화된 증적 등록
- `GET /v1/admin/benchmark-evidence`: 최근 증적 조회, 선택적으로 `release_id` 필터 적용
- `POST /v1/admin/model-releases/{id}/promote`: 모든 배치 프로필의 최신 증적을 검사하고 `ACTIVE` 승격
- `POST /v1/admin/model-releases/{id}/transition`: `ACTIVE` 요청도 동일한 승격 게이트를 통과

증적 등록자는 벤치마크 전에 context API로 중앙 노드 UUID, 현재 인벤토리 지문과 레지스트리의 고정 식별자를 조회해 측정 입력에 결합해야 합니다. 서버는 증적 등록 시 현재 저장된 프로필로 지문을 다시 계산해 불일치를 거부합니다. 현재는 이를 자동화하는 전용 CLI나 Agent 벤치마크 작업이 없으므로, 관리자 API를 직접 연동하는 신뢰된 도구만 사용할 수 있습니다.

## 목표 작업 부하

최종 자격 검증은 최소한 아래 작업 부하를 포함해야 합니다.

| 구분 | 입력/출력 | 동시성 | 목적 |
|---|---:|---:|---|
| 짧은 대화 | 1k / 128 토큰 | 1, 4, 8 | TTFT와 일반 대화 품질 |
| 긴 대화 | 4k / 256 토큰 | 1, 4, 8 | KV 캐시와 지연 시간 |
| 목표 최대 컨텍스트 | 정책별 | 1 이상 | 메모리 한계와 안정성 |
| 실제 품질 세트 | 한국어·영어·코딩·추론·도구 호출 | 정책별 | 모델 선택 품질 점수 |

현재 증적은 한 번의 구조화된 측정 요약을 저장하며, 이 전체 매트릭스의 각 조합을 서버가 직접 실행하거나 모두 수행했는지 검증하지는 않습니다. 고정된 작업 부하를 안전하게 실행하고 결과를 수집하는 기능은 Agent 벤치마크 작업과 함께 추가합니다.

## 다중 노드 검증의 남은 범위

현재 게이트는 제출된 RTT, 대역폭, 패킷 손실과 NCCL 결과를 배치 프로필의 임계값과 비교합니다. 네트워크 시험이나 NCCL 집합 연산을 직접 실행하지 않습니다.

다중 노드 파이프라인의 24시간 지속 실행, 의도적인 중지·재시작, 워커 손실, 작업 임대 재전달 복구는 아직 자동 검증하지 않습니다. 이 수용 검사는 실제 GPU 환경에서 별도로 수행해야 하며, 후속 Agent 작업과 배포 세대 검증에 연결합니다. 문서의 RTT 5ms·1Gbps 값은 초기 출발점일 뿐이고, 실제 서비스 SLO를 만족한 측정값을 배치 프로필 기준으로 사용해야 합니다.

## 결과 관리

원본 벤치마크 데이터와 큰 로그는 Git이나 중앙 증적 행에 넣지 않습니다. 접근 제어된 별도 저장소에 보관하고 중앙 제어면에는 다음 요약만 남깁니다.

- Dure 커밋, suite와 정책 버전
- 릴리스·배치 프로필과 정렬된 중앙 노드 UUID
- 아티팩트·런타임 다이제스트와 인벤토리 지문
- 작업 부하 크기와 수치 지표
- 통과·실패 상태와 실패 코드

모델 목록을 사람이 관리하는 Markdown 표로 복제하지 않습니다. 모델 레지스트리의 `DRAFT`, `VALIDATED`, `ACTIVE`, `DEPRECATED`, `REVOKED` 상태가 중앙의 진실의 원천입니다. 증적 저장과 승격 게이트는 구현되었지만 네트워크·NCCL 증적을 중앙 추천의 배치 가능성 판단에 연결하는 기능은 아직 없으므로 다중 노드 후보를 실패 안전 방식으로 거부합니다.
