# 벤치마크 및 모델 자격 검증

> 상태: **계획됨**. 모델 레지스트리와 읽기 전용 추천은 구현되었으며, 이 문서는 후보 아티팩트·런타임 조합을 증적에 따라 `VALIDATED` 또는 `ACTIVE`로 승격하기 위한 후속 기준을 정의합니다.

## 목적

모델의 크기나 공급자 벤치마크만으로 배포를 결정하지 않습니다. Dure가 실제로 사용할 GPU, 드라이버, vLLM 이미지, 네트워크, 컨텍스트, 동시성에서 측정한 품질·성능·안정성만 승격 근거로 사용합니다.

## 실행 전 고정 항목

각 벤치마크 결과에는 다음을 반드시 기록합니다.

- Dure Git 커밋과 정책·카탈로그 버전
- 모델 저장소, 변경 불가능한 리비전, 매니페스트 다이제스트, 양자화, 라이선스
- vLLM OCI 이미지 다이제스트와 런타임 버전
- GPU UUID, 모델, VRAM, 연산 능력, 드라이버·CUDA 정보
- 노드 UUID, 네트워크 인터페이스, 토폴로지, TP/PP/DP/EP 설정
- 입력 길이, 출력 길이, 동시성, 예열, 요청 수

자격 증명, 모델 토큰, 실제 사용자 프롬프트, 컨테이너 환경 변수·명령은 결과에 기록하지 않습니다.

## 기본 작업 부하

최소한 아래 작업 부하를 실행합니다.

| 구분 | 입력/출력 | 동시성 | 목적 |
|---|---:|---:|---|
| 짧은 대화 | 1k / 128 토큰 | 1, 4, 8 | TTFT와 일반 대화 품질 |
| 긴 대화 | 4k / 256 토큰 | 1, 4, 8 | KV 캐시와 지연 시간 |
| 목표 최대 컨텍스트 | 정책별 | 1 이상 | 메모리 한계와 안정성 |
| 실제 품질 세트 | 한국어·영어·코딩·추론·도구 호출 | 정책별 | 모델 선택 품질 점수 |

각 실행은 예열 20회 뒤 최소 200 요청 또는 15분 이상 측정합니다. 실제 서비스의 작업 부하가 다르면 동일한 형식으로 별도 프로필을 추가합니다.

## 수집 지표

- 콜드·웜 시작과 준비 상태 시간
- 요청 성공률, 오류, OOM, 컨테이너·Ray 재시작
- TTFT, TPOT, 종단 간 지연 시간의 p50/p95
- 단일 요청 및 전체 출력 토큰/초
- 최대·예약 VRAM, 호스트 RAM, GPU 사용률, KV 캐시 사용률
- 다중 노드의 RTT p50/p95, 대역폭, 패킷 손실, NCCL 집합 연산 결과
- `/health`, `/v1/models`, 결정론적 간이 요청 결과

## 기본 승격 통과 기준

`VALIDATED`가 되려면 다음을 통과해야 합니다.

- OOM과 프로세스 충돌이 없어야 함
- 최소 10%의 VRAM 여유가 남아야 함
- 단기 요청 성공률이 99% 이상이어야 함
- 정책의 p95 TTFT/TPOT/E2E SLO를 만족해야 함
- 승인된 기준선보다 처리량이 허용 범위 이상 하락하지 않아야 함
- 아티팩트 리비전, 모델 매니페스트, 이미지 다이제스트가 모두 고정되어야 함

다중 노드 파이프라인은 추가로 24시간 지속 실행, 중지·재시작, 워커 손실, 작업 임대 재전달 복구를 통과해야 합니다. 문서의 RTT 5ms·1Gbps 권장은 초기 출발점일 뿐이며, 실제 SLO를 만족한 측정값을 배치 프로필의 최종 기준으로 사용합니다.

## 결과 관리

원본 벤치마크 데이터와 큰 로그는 Git에 넣지 않습니다. 릴리스 아티팩트 또는 접근 제어된 저장소에 보관하고, Git에는 다음만 남깁니다.

- 재현 가능한 명령 및 설정 요약
- 커밋, 아티팩트·런타임 다이제스트, 하드웨어 지문
- 지표 요약과 통과·실패 판단
- 알려진 제한과 복구 대상

모델 목록을 사람이 수동으로 관리하는 Markdown 표로 복제하지 않습니다. 현재 레지스트리가 `DRAFT`, `VALIDATED`, `ACTIVE`, `DEPRECATED`, `REVOKED` 상태의 진실의 원천입니다. 다만 이 문서의 증적 저장과 자동 승격 게이트가 구현되기 전에는 중앙 추천기가 네트워크·NCCL 증적을 확인할 수 없으므로 다중 노드 후보를 실패 안전 방식으로 거부합니다.
