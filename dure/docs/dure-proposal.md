## 1. 한 줄 정의

개인과 조직이 제공하는 유휴 GPU를 신뢰 가능한 지역별 컴퓨팅 pod로 묶고, 기여자와 일반 사용자가 대형 언어 모델을 이용할 수 있게 하는 OpenAI 호환 커뮤니티 추론 API이다.

## 2. 제안 배경

대형 언어 모델을 직접 운영하려면 고가의 GPU와 지속적인 전력·운영 비용이 필요하다. 반면 개인 연구자, 소규모 팀, 학교와 커뮤니티에는 사용하지 않는 시간대의 GPU가 분산되어 있다. 이 자원을 안전하게 모으고 적절한 작업을 배정한다면 중앙의 대규모 GPU 서버 없이도 유용한 추론 서비스를 만들 수 있다.

본 프로젝트는 전 세계의 임의 GPU를 곧바로 하나의 거대한 GPU처럼 결합하려 하지 않는다. 초기에는 신뢰된 참여자가 제공하는 안정적인 노드를 지역별 pod로 구성하고, 모델 크기와 네트워크 조건에 따라 단일 노드 추론, 데이터 병렬 처리, 파이프라인 병렬 처리를 선택한다.

## 3. 역사적 3×24GiB 기준값과 현재 증명 범위

다음 값은 3대의 RTX 3090 24GB GPU, Ray와 vLLM으로 `Qwen2.5-72B-Instruct-AWQ`를 시험하기 위해
정리한 역사적 기준값이다. 현재 repository에는 이 실행을 source·package·model/runtime digest·node/GPU
UUID·inventory identity와 함께 재현한 `PASSED` 릴리스 수용 증적이 없다. 따라서 이 표는 현재 release의
배포 승인이나 장기 안정성 증명이 아니라 수용 검사를 설계할 때 쓰는 참고값이다.

기록된 시험 구성은 다음과 같다.

- Ray 클러스터 리소스: GPU 3개
- 모델 체크포인트: 약 38.74GiB
- 양자화 방식: AWQ 4비트, Marlin 연산 커널
- 분산 방식: Pipeline Parallel 3, Tensor Parallel 1, Data Parallel 1
- 모델 레이어 분배: 27 / 27 / 26
- 통신 백엔드: NCCL
- Attention 백엔드: FlashAttention 2
- 중간 서버의 가중치 메모리: 약 11.53GiB
- 중간 서버의 가용 KV 캐시: 약 9.23GiB
- 중간 서버 GPU 프로세스 사용량: 약 19.6GiB

이 기준값은 신뢰된 노드와 안정적인 네트워크가 있는 소규모 pod를 검증 대상으로 삼는 이유를 설명한다. 실제 수용 결과는 [릴리스 수용 검증](release-validation.md) 절차와 [v0.4.21 수용 증적](release-evidence/v0.4.21.md)을 기준으로 분리해 기록하며, 현재 상태는 `NOT_RUN`이다. 불특정 다수의 인터넷 노드로 확장하려면 장애 복구, 보안, 지연 시간과 인센티브 문제를 별도로 해결해야 한다.

## 4. 비전과 목표

### 비전

누구나 남는 GPU를 기여하고, 누구나 합리적인 범위에서 공개 언어 모델을 사용할 수 있는 커뮤니티 컴퓨팅 공공재를 만든다.

### MVP 목표

1. 신뢰된 참여자 5~20명이 노드를 등록할 수 있다.
2. OpenAI 호환 API로 채팅 및 텍스트 생성 요청을 받을 수 있다.
3. 단일 GPU 모델과 3-GPU 72B pipeline pod를 함께 운영한다.
4. 노드 상태, 모델 배치, 요청량과 기여량을 측정한다.
5. 기여자가 제공한 유효 GPU 시간에 따라 API 크레딧을 지급한다.
6. 노드 장애 시 신규 요청을 정상 pod로 우회하고, 실패한 요청을 안전하게 재시도한다.

### MVP에서 하지 않는 것

- 익명 노드를 곧바로 신뢰하는 완전 무허가 네트워크
- 임의 사양의 GPU를 실시간 tensor parallel로 결합하는 기능
- 민감정보 처리가 가능한 수준의 강한 기밀성 보장
- 상용 서비스 수준의 고가용성 SLA
- 토큰 또는 암호화폐 발행
- 모델 학습과 파인튜닝 자원 공유

## 5. 핵심 사용자

### GPU 기여자

- 사용하지 않는 시간대의 GPU를 보유한 개인 개발자
- 연구실, 학교와 커뮤니티 서버 운영자
- 유휴 워크스테이션을 보유한 소규모 기업

기여자는 GPU 시간과 안정성을 제공하고 API 크레딧, 기여 통계와 커뮤니티 권한을 받는다.

### API 사용자

- 공개 모델을 실험하려는 개발자와 학생
- 비용에 민감하고 요청 지연을 일부 감수할 수 있는 프로젝트
- 비동기 생성, 평가, 데이터 가공이 필요한 연구자

### 후원자

- 공공 AI 인프라를 지원하려는 기업, 학교와 비영리 조직
- 무료 사용자용 공용 크레딧 풀에 예산이나 하드웨어를 제공하는 주체

## 6. 제품 원칙

1. **로컬 우선:** 분산 pod는 가능한 한 같은 LAN 또는 낮은 지연 시간의 지역 안에서 구성한다.
2. **작업에 맞는 분산:** 작은 모델은 단일 GPU, 다수 요청은 데이터 병렬, 큰 모델은 안정적인 pod의 파이프라인 병렬로 처리한다.
3. **제어면과 데이터면 분리:** 공개 API가 Ray GCS나 worker 포트에 직접 접근하지 못하게 한다.
4. **검증 가능한 기여만 보상:** 등록 시간이 아니라 실제 완료한 유효 GPU 작업을 기준으로 크레딧을 계산한다.
5. **프라이버시 수준을 명시:** 보장할 수 없는 기밀성을 과장하지 않고 요청별 처리 등급을 공개한다.
6. **무료는 무제한이 아님:** 무료 사용량은 기여, 후원과 운영 예산 안에서 지속 가능한 수준으로 제한한다.

## 7. 서비스 구성

> **미구현 설계:** 이 절의 API Gateway, Scheduler, 사용자 요청 routing·metering은 장기 제품 설계이며
> 현재 Control Plane이나 Agent가 제공하지 않습니다.

```text
API 사용자
    │ HTTPS / OpenAI-compatible API
    ▼
API Gateway ── 인증, 쿼터, 속도 제한, 스트리밍
    │
    ▼
Scheduler ── 모델, 지역, 지연, GPU, 신뢰도 기준으로 pod 선택
    │
    ├── Single-GPU Pool ── 7B~32B 모델, 데이터 병렬
    │
    ├── Regional 72B Pod ── 2~4개의 신뢰된 노드, pipeline parallel
    │
    └── Batch Pool ── 비동기 생성, 평가, 임베딩

Control Plane
    ├── Node Registry / Heartbeat
    ├── Model Registry
    ├── Usage Metering / Credit Ledger
    ├── Health Check / Benchmark
    └── Metrics / Logs / Audit
```

### API 게이트웨이

- API 키와 사용자 인증
- 무료 쿼터 및 기여 크레딧 차감
- 요청 크기와 동시성 제한
- OpenAI 호환 엔드포인트 제공
- 스트리밍 응답 중계
- 요청 ID 기반 추적 및 재시도 제어

초기 지원 엔드포인트는 다음으로 제한한다.

- `GET /v1/models`
- `POST /v1/chat/completions`
- `POST /v1/completions`(필요 시)
- `POST /v1/batches`(2단계)

### 스케줄러

각 요청에 대해 다음 정보를 종합해 실행 위치를 선택한다.

- 요청 모델과 양자화 버전
- 모델이 이미 캐시된 노드인지 여부
- GPU VRAM, 성능 등급과 현재 사용률
- 노드 간 RTT, 대역폭과 지역
- 최근 성공률과 가동 시간
- 예상 대기 시간과 요청 우선순위
- 개인정보 처리 등급

한 대화의 KV 캐시는 특정 pod에 있으므로, 진행 중인 세션은 가능한 한 같은 pod에 고정한다. 해당 pod가 실패하면 KV 캐시는 유실될 수 있으며 MVP에서는 전체 요청 또는 마지막 턴을 다른 pod에서 재실행한다.

### 노드 에이전트

기여자 컴퓨터에 설치되는 경량 에이전트이다.

- GPU, 드라이버, VRAM과 네트워크 벤치마크
- 서명된 컨테이너 이미지 실행
- 모델 파일 다운로드 및 해시 검증
- 노드 heartbeat와 작업 상태 보고
- GPU 시간, 완료 토큰과 오류 측정
- 사용 가능 시간대와 자원 상한 설정
- 즉시 참여 중단 기능

MVP에서는 Linux와 NVIDIA GPU만 지원한다. 분산 72B pod는 우선 24GB VRAM 이상, 검증된 드라이버 조합, 안정적인 유선 네트워크를 요구한다.

## 8. 추론 전략

> **미구현 확장:** data parallel, 인터넷 환경의 batch pool, 자동 pod 선택은 현재 지원하지 않습니다.
> 실제 runtime은 지원 매트릭스의 단일 GPU 및 제한된 `TP=1`·`PP=2/3` 계약만 따릅니다.

### 단일 GPU 풀

GPU 한 장에 들어가는 양자화 모델을 각 노드가 독립적으로 실행한다. 동일 모델을 여러 노드에 복제하고 요청을 분산하므로 장애 격리와 확장이 가장 쉽다. MVP의 기본 경로로 사용한다.

### 데이터 병렬 풀

각 GPU가 모델 전체를 보유하고 서로 다른 요청을 처리한다. 노드 사이에 토큰별 통신이 필요하지 않아 인터넷 환경에 가장 적합하다.

### 지역별 72B 파이프라인 포드

한 모델의 레이어를 2~4개 구간으로 나누고 각 GPU가 한 구간을 담당한다. 단계 사이에서는 NCCL을 통해 activation을 전달한다. 전체 가중치를 한 GPU에 적재할 필요는 없지만, 각 토큰이 모든 단계를 지나야 하므로 낮고 안정적인 네트워크 지연이 중요하다.

초기 72B pod의 잠정 조건은 다음과 같다.

- 동일하거나 유사한 GPU 등급
- GPU당 VRAM 24GB 이상
- pod 내부 유선 네트워크 1Gbps 이상 권장
- pod 내부 RTT 5ms 이하 권장
- 모델 및 vLLM 버전 고정
- 노드 운영자를 사전에 승인
- pod마다 최소 하나의 대체 실행 경로 확보

### 비동기 배치 풀

응답 지연에 민감하지 않은 작업을 불안정하거나 느린 노드에 배정한다. 실시간 서비스보다 노드 변동에 강하므로 공개 참여 범위를 넓히는 첫 수단으로 사용한다.

## 9. 장애 처리

> **미구현 자동화:** 자동 요청 우회·streaming 재시도·stage 교체는 현재 기능이 아닙니다. 현재는
> 명시적 retry, node 격리, 검증된 직전 generation rollback만 운영 절차로 지원합니다.

분산 추론에서는 한 단계의 장애가 전체 요청 실패로 이어진다. MVP에서는 복잡한 실시간 stage 교체보다 단순하고 예측 가능한 복구를 우선한다.

- heartbeat 누락 시 신규 요청 배정 즉시 중단
- pipeline stage 장애 시 현재 요청 실패 처리 후 정상 pod에서 재시도
- 스트리밍 시작 전 실패는 자동 재시도
- 스트리밍 도중 실패는 명확한 오류 이벤트 반환
- 반복 실패 노드는 격리 후 재벤치마크
- 모델 해시와 런타임 버전이 다른 노드는 pod 참여 금지
- 동일 모델의 예비 pod 또는 단일 노드 대체 모델 운영

향후에는 prefix/KV 캐시 복제, stage replica와 checkpoint 기반 복구를 검토한다.

## 10. 보안과 개인정보 보호

### 기본 보안

- 노드와 제어면 사이 mTLS 인증
- WireGuard 등의 사설 오버레이 네트워크
- Ray GCS, dashboard 및 worker 포트의 공개 인터넷 노출 금지
- rootless 또는 강하게 제한된 컨테이너
- 읽기 전용 이미지와 서명 검증
- 호스트 파일시스템 및 Docker socket 접근 금지
- 기본 outbound 네트워크 차단과 목적지 allowlist
- 모델 및 실행 이미지의 SHA-256 해시 검증
- 중앙 비밀값을 worker에 전달하지 않는 구조

### 프롬프트 보호

전송 구간은 암호화할 수 있지만 일반 소비자 GPU에서 계산하는 동안 운영자에게 내용을 완전히 숨기는 것은 어렵다. 따라서 MVP는 다음 등급을 명시한다.

- **Community:** 비민감 요청 전용, 신뢰된 커뮤니티 노드 처리
- **Private:** 중앙 또는 별도 승인된 전용 노드에서만 처리
- **No-log:** Gateway와 worker의 프롬프트 로그 저장 금지, 메타데이터만 기록

주민번호, 의료정보, 인증정보, 영업비밀 등 민감정보는 Community 등급에서 처리하지 않도록 약관과 UI에 명시한다.

### 악성 또는 오류 노드 대응

- 고정된 canary 요청을 통한 결과 이상 탐지
- 모델 및 런타임 버전 attestation
- 일부 작업의 중복 실행과 결과 비교
- 비정상 latency, 오류율과 출력 분포 감시
- 신뢰 점수 하락 시 실시간 풀에서 배치 풀로 강등
- 반복 위반 시 인증서 폐기와 크레딧 몰수

LLM 출력은 비결정적일 수 있으므로 단순 문자열 일치만으로 계산 정합성을 검증하지 않는다. 결정적 설정의 canary와 내부 activation/checksum 검증을 단계적으로 도입한다.

## 11. 기여 보상과 무료 정책

> **미구현 경제 기능:** usage meter, credit ledger, quota, 기여 보상은 현재 저장·계산·집행하지 않습니다.

### 기본 모델

기여자가 실제로 완료한 유효 GPU 작업을 크레딧으로 환산한다. 암호화폐 없이 내부 원장으로 시작한다.

```text
기여 크레딧 = 유효 GPU 초
             × GPU 성능 계수
             × 신뢰도 계수
             × 수요 계수
```

- GPU 성능 계수: 표준 벤치마크 처리량과 VRAM 기준
- 신뢰도 계수: 가동 시간, 성공률, 중도 이탈률 기준
- 수요 계수: 현재 부족한 모델·지역·시간대에 가중치 부여
- 유효 GPU 초: 모델 준비나 실패 시간이 아니라 완료된 작업 중심

### 사용자 정책

- 신규 사용자에게 소량의 일일 무료 쿼터 지급
- 기여자는 획득 크레딧 범위에서 우선 사용
- 무료 사용자는 대기열과 낮은 동시성 적용
- 후원 크레딧은 교육·오픈소스 프로젝트에 배분
- 장기적으로 유료 크레딧을 도입하더라도 기여 크레딧과 공용 무료 풀 유지

### 남용 방지

- 이메일 또는 OAuth 기반 계정 인증
- 사용자·API 키·IP별 속도 제한
- 모델별 입력 및 출력 토큰 상한
- 비정상적인 다계정 생성 탐지
- 공개 데모와 API 쿼터 분리
- 채굴, 무한 agent loop 등 과도한 자동화 제한

## 18. 검증 실험

MVP 개발 전후에 다음 실험을 반복한다.

1. **네트워크 지연 실험:** pod 단계 사이에 인위적 RTT를 추가해 TTFT와 tokens/s 변화를 측정한다.
2. **노드 장애 실험:** 생성 중 한 worker를 종료하고 실패 감지와 재시도 동작을 확인한다.
3. **GPU 이질성 실험:** 서로 다른 GPU를 한 pod에 넣었을 때 병목과 메모리 불균형을 측정한다.
4. **동시 요청 실험:** continuous batching의 처리량과 pipeline bubble 감소 효과를 측정한다.
5. **모델별 비교:** 7B 단일 GPU, 32B 단일/분산, 72B pipeline의 비용과 지연을 비교한다.
6. **프라이버시 점검:** Gateway, worker, Ray와 시스템 로그에 프롬프트가 남는지 확인한다.
7. **보상 검증:** 단순 접속 시간이 아닌 완료된 유효 작업량과 크레딧이 일치하는지 확인한다.

## 19. 필요한 초기 역할

- 분산 추론/인프라 엔지니어: vLLM, Ray, GPU와 네트워크
- 백엔드 엔지니어: Gateway, Scheduler, 사용량 원장
- 보안 엔지니어 또는 리뷰어: Agent 격리, mTLS, 위협 모델
- 제품/커뮤니티 운영: 기여자 모집, 정책과 지원

초기에는 2~3명이 여러 역할을 겸할 수 있지만, 공개 노드를 받기 전에는 반드시 별도의 보안 리뷰가 필요하다.

## 20. 의사결정 기준

12주 알파 종료 시 다음 질문으로 다음 투자를 결정한다.

1. 공급자가 반복적으로 GPU를 제공하는가?
2. 무료 사용자 수요가 실제 기여와 후원 범위 안에서 유지되는가?
3. 72B pipeline pod가 단일 고성능 호스팅 대비 의미 있는 비용 또는 접근성 이점을 제공하는가?
4. 장애와 네트워크 변동 속에서도 허용 가능한 사용자 경험을 제공하는가?
5. 프롬프트와 호스트를 보호할 현실적인 보안 경계를 유지할 수 있는가?

세 가지 이상이 부정적이라면 거대 모델의 실시간 분산보다 단일 GPU 모델의 데이터 병렬 API 또는 비동기 배치 컴퓨팅에 집중한다.

## 21. 결론

이 프로젝트의 경쟁력은 단순히 여러 GPU를 연결하는 기술에 있지 않다. 서로 다른 소유자의 자원을 신뢰도, 위치, 모델 적합성에 따라 묶고, 유효한 기여를 측정해 지속 가능한 무료 사용으로 전환하는 운영 체계가 핵심이다.

현재의 3×RTX 3090 Qwen2.5-72B pod는 수용 검사를 위한 첫 번째 **기준 가설**이다. 실제 GPU 환경의 `PASSED` 증적과 24시간 복구 검증이 축적되기 전에는 안정적인 공급 단위 또는 현재 release의 성공 증명으로 표현하지 않는다. 다음 단계에서는 이 기준을 격리된 수용 검사로 재현하고, 단일 GPU pool과 제어면을 구축한 뒤 소수의 신뢰된 기여자를 연결해야 한다. 이 순서라면 과도한 탈중앙화 약속 없이도 실제 사용 가능한 커뮤니티 LLM API를 검증할 수 있다.
