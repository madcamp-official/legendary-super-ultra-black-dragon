# 벤치마크 및 모델 자격 검증

> 상태: **부분 구현**. 중앙 제어면의 벤치마크 증적 저장, 고정 식별자 검증, SLO 판정, `ACTIVE` 승격 게이트, 정확한 노드 조합의 최신 증적을 소비하는 읽기 전용 중앙 추천과 승인된 단일 GPU 노드의 폐쇄형 Agent 작업은 구현되었습니다. 중앙 아티팩트 캐시 수명 주기와 배포의 exact `READY` 소비 게이트도 구현되었지만, 자동 벤치마크는 아직 단일 GPU의 `FULL_SNAPSHOT` 캐시와 `dure-benchmark` 컨테이너 계약만 지원합니다. vLLM 단계 아티팩트의 export/load 증적은 별도 레지스트리에서 관리하며 모델 릴리스 자격 증적을 대신하지 않습니다. 전체 작업 부하 매트릭스, 다중 노드 네트워크·NCCL 시험 자동 실행, 실제 GPU stage 배포와 24시간 복구 검증은 후속 범위입니다.

## 현재 제공 범위

중앙 제어면은 관리자 인증을 거친 구조화된 증적만 저장합니다. 증적 등록 자체는 Agent 작업, 모델 다운로드, 이미지 내려받기, Docker 실행 또는 배포 변경을 만들지 않습니다.

- 고정 suite `dure-serving-slo-v1`과 정책 `benchmark-gate-v1`만 받습니다.
- 모델 릴리스, 배치 프로필, 정렬된 중앙 노드 UUID, 현재 노드 프로필 지문을 하나의 증적에 결합합니다.
- 모델 아티팩트 리비전·매니페스트 다이제스트와 OCI 런타임 이미지가 레지스트리 값과 정확히 같아야 합니다.
- 운영자가 제출한 Dure 커밋 표식, 입력·출력 토큰 수, 동시성, 예열·측정 요청 수와 측정 시간을 기록합니다.
- 품질, 지연 시간, 처리량, 성공률, VRAM 여유, OOM·충돌·재시작, 네트워크와 NCCL 결과를 형식화된 필드로 저장합니다.
- 서버가 판정한 `PASSED` 또는 `FAILED` 상태와 고정된 실패 코드를 저장합니다. SLO 미달 결과도 버리지 않고 `FAILED` 증적으로 남깁니다.
- 자동 실행의 같은 run 재보고와 수동 등록의 연속된 동일 재전송은 멱등 처리합니다. 다른 결과가 등록되거나 최신 수동 증적 뒤의 자동 실행이 실패한 다음 과거와 같은 수동 값이 다시 제출되면 새 측정 발생으로 보고 새 등록 순번을 부여하며, 배치 프로필별 가장 큰 순번으로 최신 결과를 판정합니다.
- 중앙 다중 노드 추천은 정확히 정렬된 노드 UUID 조합별 최신 증적을 읽고, 현재 릴리스·배치·인벤토리와 일치하는 24시간 이내 `PASSED` 결과만 후보 적격성에 사용합니다.

증적 API는 프롬프트, 자격 증명, 모델 접근 토큰, 로그 본문, 임의 명령, Docker 인자, 환경 변수, 마운트, 호스트 경로 또는 자유 형식 metadata 필드를 받지 않습니다.

## 자동 Agent 실행 경로

자동 실행은 준비와 적용을 분리합니다.

1. 관리자가 정규 UUID인 `request_id`, 릴리스, 단일 노드 배치 프로필, 노드 UUID, 작업 부하와 Dure 커밋 표식으로 실행을 준비합니다.
2. 서버는 현재 인벤토리 지문, 모델 리비전·매니페스트, 양자화와 런타임 이미지 다이제스트를 `PREPARED` 실행에 고정합니다. 이 단계는 Agent 작업, Docker 실행, 다운로드나 이미지 내려받기를 만들지 않습니다.
3. 운영자가 별도 요청 본문 `{"apply": true}`를 보내면 서버가 문맥을 다시 검사하고 해당 노드에 `BENCHMARK` 작업 하나를 만듭니다. Agent 0.3.25 이상에서는 `prepare_model`과 `pull_image`를 strict boolean으로 함께 승인할 수 있고, `dure admin activate --apply`는 둘을 true로 고정합니다. 같은 `request_id`에 다른 승인값을 재전송하면 충돌로 거부합니다. 새 측정에는 새 `request_id`를 사용합니다.
4. 서버는 선택한 노드의 건강한 GPU·최소 VRAM·남은 디스크·Docker/NVIDIA 런타임이 배치 프로필을 만족하는지 준비·적용·승격 시점마다 검사합니다. 실행기가 고르는 것과 같은 규칙으로 가장 큰 정상 GPU를 선택하고, compute capability를 `8.0/8.6/8.7=ampere`, `8.9=ada`, `9.0=hopper`, `10.0/10.3/11.0/12.0/12.1=blackwell`의 폐쇄형 목록으로 해석해 런타임의 `gpu_architectures`와 비교합니다. capability가 없거나 목록 밖이거나 런타임과 불일치하면 거부합니다. Agent는 `apply=true`, coordinator와 노드 UUID, 단일 노드 조건과 안정적인 하드웨어·런타임 자격 필드로 만든 현재 프로필 지문을 다시 검증합니다. 남은 메모리·디스크, 현재 작업과 준비 과정에서 생기는 exact cache 관측은 지문에서 제외하고, 실행 직전 repository·revision·manifest·양자화 marker로 모델 자체를 별도 검증합니다.
5. 준비 승인이 없으면 실행기는 로컬에 이미 존재하는 OCI 다이제스트 이미지와 `/var/lib/dure/models` 아래의 완전히 펼쳐진 Dure 모델 캐시만 사용합니다. 준비 승인이 있으면 Agent root 전용 설정의 신뢰 HTTPS origin에서 정규 manifest 바이트를 검증해 캐시를 만들고 digest 이미지를 pull·재검사합니다. 다른 활성 작업 부하가 있으면 실행을 거부하며 Hugging Face hub snapshot을 직접 실행 입력으로 사용하지 않습니다.
6. 고정 이미지는 `dure-benchmark` 진입점을 제공해야 합니다. 실행 직전 `nvidia-smi`로 선택 GPU의 compute process가 없음을 다시 확인하고, MIG process는 상위 GPU를 안전하게 식별할 수 없으므로 하나라도 있으면 거부합니다. 컨테이너는 가장 큰 정상 GPU 한 장만 UUID로 할당하고, 읽기 전용 루트, 네트워크 없음, capability 제거, 권한 상승 금지, 비 root 사용자, `restart=no`와 고정 Dure 벤치마크 레이블로 실행됩니다. 메모리는 `min(전체 RAM, 가용 RAM) / 2`와 32GiB 중 작은 값으로 제한하고 8GiB 미만이면 실행하지 않으며, `memory-swap`을 같은 값으로 두어 추가 swap을 막습니다. CPU quota는 논리 CPU 수의 절반과 8코어 중 작은 값입니다. stdout·stderr는 실행 중 합계 64KiB로 제한하고 중앙에는 원문을 보내지 않습니다.
7. Agent와 서버가 동일한 폐쇄형 결과 스키마와 유한 수치 범위를 검사합니다. 성공 결과는 자동으로 증적에 등록되고, SLO 미달도 정상 수집된 `FAILED` 증적으로 남습니다.

`POST /v1/admin/benchmark-runs/prepare`는 모델 바이트를 준비하는 API가 아니라 DB에 고정 실행 문맥만 저장합니다. 실제 모델·이미지 변경은 별도 apply에서 `prepare_model=true`, `pull_image=true`를 명시했을 때만 일어납니다. 이 자격 검증용 준비는 benchmark 실행 전 exact local 입력을 만들지만 추천 세대의 중앙 `READY` 증적을 대신하지 않습니다. 승격·추천 수락 뒤 deployment prepare가 같은 콘텐츠를 재검증하고 현재 준비 시도에 결합된 `READY`를 만들어야 실제 배포가 시작됩니다.

## 아티팩트 캐시 게이트와 실패 대응

추천으로 만든 배포의 apply·start·restart·verify와 rollback target 시작은 모든 대상 노드에서 선택된 exact 캐시가 현재 준비 시도에 결합된 `READY`이고 최신 이미지 digest 준비 증적이 있을 때만 작업을 만듭니다. `STALE`·`MISSING`·`CORRUPT`·`QUARANTINED`, 철회된 stage variant, 이전 시도의 늦은 보고나 이미지 drift는 실행 전에 거부됩니다. 추천과 수락 자체는 여전히 다운로드, pull, 배포 작업을 만들지 않습니다.

자동 `BENCHMARK`는 배포 backend를 선택하는 경로가 아니며, 단일 GPU의 폐쇄형 `dure-benchmark` 컨테이너만 실행합니다. Agent는 실행 직전에 repository·revision·manifest·양자화·verification version이 모두 일치하는 Dure `FULL_SNAPSHOT` 캐시가 딱 하나인지 다시 검사합니다. `STAGE` 캐시, `VLLM_RAY_PP_V1` 다중 노드 backend, 다중 GPU 위치는 이 실행기가 지원하지 않으며 잘못된 조합은 `BENCHMARK_ARTIFACT_UNAVAILABLE` 또는 `MULTI_NODE_BENCHMARK_UNSUPPORTED`로 닫힙니다. 현재 중앙 benchmark apply는 `node_artifact_caches` 행 자체를 실행 증거로 소비하지 않으며, Agent의 exact 로컬 캐시·marker·이미지 재검사가 실행 게이트입니다.

`dure admin artifact-cache verify <cache-id>`는 중앙 캐시 행과 참조 투영만 읽는 조회이며, 파일 전체 재해시나 벤치마크를 실행하지 않습니다. `artifact-cache quarantine`도 기본은 task 0개인 preview입니다. `--apply`는 현재·직전 세대, 활성 준비·배포·벤치마크 작업과 컨테이너 mount가 참조하지 않는 exact final 하나만 보존 영역으로 원자 이동합니다. 자동 삭제·퇴출·복구는 제공하지 않으며, 오염된 캐시는 격리한 뒤 현재 준비를 새로 성공시켜야 `READY`로 복구됩니다.

작업 페이로드는 임의 명령, Docker 인자, 환경 변수, 마운트, 호스트 경로, Python 코드, 프롬프트, 토큰, 비밀값 또는 로그 필드를 허용하지 않습니다. 실패 시 중앙 DB와 감사 이벤트에는 고정 실패 코드만 저장하고 원문 stdout·stderr·예외문은 저장하지 않습니다.

콘텐츠 주소 준비기가 만드는 자동 실행용 모델 디렉터리에는 다음 v2 `.dure-model.json`이 있어야 합니다. Agent와 실행기는 이 값을 준비된 레지스트리 식별자와 정확히 비교하고 `FULL_SNAPSHOT`, 검증 버전 1만 허용합니다. `STAGE` 캐시를 전체 모델로 오인해 단일 GPU 벤치마크에 사용하지 않습니다.

```json
{
  "schema": "dure-model-cache-v2",
  "repository": "Example/Model-AWQ",
  "revision": "<40~64자리 커밋 해시>",
  "manifest_digest": "sha256:<64자리 해시>",
  "quantization": "awq",
  "cache_kind": "FULL_SNAPSHOT",
  "verification_version": 1
}
```

v1 marker는 기존 캐시 읽기 호환성을 위해 `FULL_SNAPSHOT`, 검증 버전 1로 해석합니다. 새 준비기는 v2만 기록하며 청크·파일 전체 SHA-256, exact tree와 `config.json` 양자화 일치를 확인한 뒤 marker를 마지막에 씁니다. 그러나 v2 형식 자체가 준비기 실행을 증명하지는 않습니다. 벤치마크는 매니페스트 문서를 받지 않아 실행 직전에 모든 모델 파일을 다시 해시하지 않으므로, 보호된 Dure 루트와 중앙 준비 증거가 없는 수동 v1 또는 v2 marker를 새 검증 캐시로 간주해서는 안 됩니다. 어느 형식도 모델 게시자의 암호학적 서명이나 provenance를 증명하지 않습니다.

재전달된 같은 작업은 폐쇄형 payload의 명시적 적용·coordinator·단일 노드 조건을 먼저 확인한 다음, 현재 Agent 빌드·프로필 지문·모델 캐시·가용 자원·이미지를 검사하기 전에 정확한 이름·레이블·Docker 시작 시각을 조정합니다. 따라서 실행 중 Agent가 업그레이드되거나 캐시·가용 메모리가 달라져도 고아 컨테이너 정리가 먼저 진행됩니다. 조정 뒤에는 프로필을 새로 조사하고 원래 실행 조건을 모두 다시 검사합니다. 실행 시간이 고정 측정 900초와 정리 여유 300초를 넘지 않은 `running`·`restarting`·`paused` 컨테이너라면 새 실행이나 실패 확정을 하지 않고 임대 재시도를 기다립니다. 1,200초를 넘긴 정확한 컨테이너와 같은 작업의 `created`·`exited`·`dead` 컨테이너만 레이블을 다시 확인한 뒤 중지·제거하고 재시도합니다. inspect·stop·remove 결과나 부재를 확증할 수 없으면 중앙에 terminal 실패를 보내지 않고 재시도를 유예합니다. 레이블·시작 시각이 유효하지 않은 이름 충돌은 손대지 않고 실패합니다. 다른 Dure 벤치마크 컨테이너는 상태와 관계없이 병행 실행을 차단하며 자동으로 제거하지 않습니다. 운영자는 원인을 확인한 뒤 해당 컨테이너를 별도로 정리해야 합니다.

GPU process 확인과 Docker 시작은 하나의 원자 연산이 아니므로 두 시점 사이에 다른 로컬 process가 GPU를 점유하는 경주는 남아 있습니다. 벤치마크 노드는 신뢰된 전용 운영 경계에서 사용하고, Dure 밖의 GPU 스케줄러가 동시에 작업을 시작하지 않게 해야 합니다.

공식 Debian 패키지는 빌드 커밋을 `/usr/share/dure/build-commit`에 읽기 전용 metadata로 설치합니다. Agent는 새 실행 전에 `dure_commit`이 현재 설치 값과 정확히 일치해야만 프로필 조사와 컨테이너 실행을 진행하고, 완료 이력에도 실제 실행 커밋을 함께 저장합니다. 완료 보고가 실패한 뒤 Agent가 업그레이드돼도 이미 검증된 성공 이력의 실행 커밋이 작업 payload와 같고 결과 스키마가 유효하면 재실행하지 않고 같은 성공을 다시 보고합니다. 일반 wheel이나 editable 설치에는 이 파일이 없으므로 신뢰된 개발 환경에서 `DURE_BUILD_COMMIT`을 명시하지 않으면 새 벤치마크만 안전하게 거부됩니다. 이 결합은 잘못된 커밋 표식을 막지만 결과의 암호학적 서명이나 원본 provenance 증명을 대신하지는 않습니다.

## 증적과 릴리스 상태

배치 프로필은 `DRAFT` 릴리스에만 추가할 수 있습니다. 프로필 구성을 마친 뒤 릴리스를 `VALIDATED`로 전환하고 외부에서 측정한 증적을 등록합니다.

`VALIDATED → ACTIVE` 승격에는 다음 조건이 모두 필요합니다.

1. 릴리스의 모든 배치 프로필에 증적이 있어야 합니다.
2. 각 배치 프로필에서 등록 순번이 가장 큰 최신 증적이 `PASSED`여야 합니다. 그 증적 뒤의 자동 실행이 큐에 있거나 실행 중이면 승격을 기다리고, 실행·런타임·아티팩트 오류 또는 취소로 종료됐다면 과거 통과 결과로 우회하지 않습니다.
3. 증적의 아티팩트 리비전, 모델 매니페스트와 런타임 이미지가 현재 레지스트리 값과 같아야 합니다.
4. 증적에 결합된 노드가 계속 승인 상태이고, 현재 프로필 지문이 측정 당시 지문과 같아야 합니다.

승격에 사용한 증적 ID 집합과 그 다이제스트는 모델 릴리스에 고정합니다. 같은 릴리스의 승격을 반복 요청하면 새 상태 전이나 감사 이벤트를 만들지 않고 최초 증적 집합을 반환합니다. `ACTIVE` 이후 새 실패 증적이 등록돼도 자동 강등하거나 실행 중인 배포를 변경하지 않습니다. 재검증 실패에 따른 `DEPRECATED` 또는 `REVOKED` 전환은 운영자가 별도로 판단해야 합니다.

## 중앙 추천의 다중 노드 증적 자격

`ACTIVE` 승격과 중앙 추천은 증적을 사용하는 목적이 다릅니다. 승격은 릴리스의 모든 배치 프로필이 자격 기준을 통과했는지 확인하고, 추천은 지금 선택하려는 정확한 노드 조합이 여전히 해당 배치를 실행할 수 있는지 읽기 전용으로 다시 확인합니다.

중앙 추천은 다음 조건을 모두 만족할 때만 다중 노드 후보를 통과시킵니다.

1. 증적의 정렬된 중앙 노드 UUID 집합이 후보의 전체 노드 집합과 정확히 같습니다. 다른 증적의 노드를 합치거나 부분 집합을 재사용하지 않습니다.
2. 같은 모델 릴리스와 배치 프로필, 아티팩트 리비전·매니페스트, 런타임 릴리스·OCI 이미지에 결합된 증적입니다.
3. 현재 노드 프로필로 다시 계산한 안정 인벤토리 지문이 측정 당시 지문과 같습니다.
4. 같은 노드 조합에서 등록 순번이 가장 큰 최신 증적이 `PASSED`이고 등록 시각이 미래가 아니며 24시간 이내입니다.
5. 증적 이후에 준비됐거나 큐·실행 중인 벤치마크 실행 또는 실패한 실행이 없습니다. 새 실행 결과가 확정되기 전에는 과거 통과 결과를 사용하지 않습니다.
6. 측정 대역폭, RTT, 패킷 손실과 필요한 NCCL 집합 연산 결과가 현재 배치 프로필 임계값을 모두 만족합니다.

하나라도 누락·만료·불일치하면 후보를 실패 안전 방식으로 거부합니다. 이 조회는 추천·정규화 인벤토리 스냅샷 외에 벤치마크 실행, Agent 작업, 모델 다운로드, 이미지 내려받기, Docker 또는 호스트 상태를 만들거나 바꾸지 않습니다. 추천 수락도 적용 전 배포 세대만 만들며 다운로드·pull·작업을 시작하지 않습니다.

## 현재 승격 판정

현재 정책은 다음 조건을 평가합니다.

- 예열 요청 20회 이상
- 측정 요청 200회 이상 또는 측정 시간 15분 이상
- OOM, 프로세스 충돌과 예기치 않은 런타임 재시작 없음
- 품질 점수 0.80 이상
- 배치 프로필의 p95 TTFT, TPOT, 종단 간 지연 상한 만족
- 배치 프로필의 최소 처리량, 요청 성공률과 VRAM 여유 만족
- 다중 노드 배치의 최소 대역폭, 최대 RTT와 패킷 손실 상한 만족
- NCCL이 필요한 배치의 집합 연산 증적 존재 및 성공

누락과 실패는 OOM, 충돌, 예열·측정량, 품질, TTFT·TPOT·종단 간 지연, 처리량, 성공률, VRAM, 네트워크, NCCL 범주별 코드로 구분합니다. 수치 필드는 NaN이나 무한대를 허용하지 않습니다.

## 관리자 API

- `POST /v1/admin/benchmark-context`: 릴리스·배치 프로필·노드 UUID를 검증하고 현재 인벤토리 지문과 고정 식별자 조회
- `POST /v1/admin/benchmark-runs/prepare`: 단일 노드 실행 문맥 준비. 작업과 호스트 변경 없음
- `POST /v1/admin/benchmark-runs/{request_id}/apply`: `apply=true`일 때만 폐쇄형 작업 생성. Agent 0.3.25 이상에서는 선택적으로 `prepare_model=true`, `pull_image=true` 승인 가능
- `GET /v1/admin/benchmark-runs/{request_id}`: 준비·큐·성공·실패 상태와 연결된 작업·증적 조회
- `POST /v1/admin/benchmark-evidence`: 구조화된 증적 등록
- `GET /v1/admin/benchmark-evidence`: 최근 증적 조회, 선택적으로 `release_id` 필터 적용
- `POST /v1/admin/model-releases/{id}/promote`: 모든 배치 프로필의 최신 증적을 검사하고 `ACTIVE` 승격
- `POST /v1/admin/model-releases/{id}/transition`: `ACTIVE` 요청도 동일한 승격 게이트를 통과

수동 증적 등록자는 벤치마크 전에 context API로 중앙 노드 UUID, 현재 인벤토리 지문과 레지스트리의 고정 식별자를 조회해 측정 입력에 결합해야 합니다. 서버는 등록 시 현재 저장된 프로필로 지문을 다시 계산해 불일치를 거부합니다. 자동 경로는 같은 결합을 준비·적용 시점과 Agent 실행 직전에 반복 검사합니다. 전체 단일 GPU 활성화는 `dure admin activate`로 실행하며, 개별 벤치마크 API도 계속 사용할 수 있습니다.

## 자동 작업 부하와 남은 목표

자동 작업은 다음 네 가지 ID와 수치만 허용합니다. 예열 20회, 측정 요청 200회, 목표 측정 시간 900초는 모든 작업에 고정됩니다.

| 구분 | 입력/출력 | 동시성 | 목적 |
|---|---:|---:|---|
| `short-chat-1k-128` | 1,024 / 128 토큰 | 8 | TTFT와 일반 대화 처리량 |
| `long-chat-4k-256` | 4,096 / 256 토큰 | 4 | KV 캐시와 지연 시간 |
| `max-context` | 레지스트리 최대 길이에서 출력 256을 제외 / 256 토큰 | 1 | 로컬 모델 설정과 최대 문맥 한계 |
| `quality-eval` | 1,024 / 256 토큰 | 1 | 고정 평가기의 품질 점수 |

실행기는 `config.json`의 로컬 최대 문맥과 준비된 수치를 다시 비교합니다. 한 실행은 한 작업 부하만 측정하며, 서버가 네 작업을 자동으로 묶거나 짧은·긴 대화의 동시성 1·4·8 전체 조합을 모두 수행했는지 증명하지는 않습니다. 최종 자격 검증의 전체 매트릭스와 실제 한국어·영어·코딩·추론·도구 호출 품질 세트 확대는 후속 범위입니다.

## 다중 노드 검증의 남은 범위

현재 게이트는 외부에서 제출한 RTT, 대역폭, 패킷 손실과 NCCL 결과를 배치 프로필의 임계값과 비교하고, 조건을 통과한 정확한 노드 조합의 최신 증적을 중앙 추천에서 소비합니다. 그러나 Dure가 다중 노드 네트워크 시험이나 NCCL 집합 연산을 직접 실행하지는 않습니다. 운영자는 현재 각 노드에 이미 존재하는 모델과 런타임으로 별도 시험을 수행한 뒤 구조화된 증적을 등록해야 합니다.

다중 노드 파이프라인의 stage별 모델 아티팩트 배포, 24시간 지속 실행, 의도적인 중지·재시작, 워커 손실, 작업 임대 재전달 복구는 아직 자동 검증하지 않습니다. 0.3.12의 전체 노드 `VERIFY` 성공은 세대의 `verified_at` 롤백 증거를 만들 수 있지만 네트워크·NCCL 측정이나 24시간 복구 수용 검사를 새로 실행하지 않습니다. 이 수용 검사는 실제 GPU 환경에서 별도로 수행해야 합니다. 추천 증적의 24시간 TTL은 네트워크 측정의 신선도 제한이며 24시간 지속 실행을 검증했다는 의미가 아닙니다. 문서의 RTT 5ms·1Gbps 값은 초기 출발점일 뿐이고, 실제 서비스 SLO를 만족한 측정값을 배치 프로필 기준으로 사용해야 합니다.

## 결과 관리

원본 벤치마크 데이터와 큰 로그는 Git이나 중앙 증적 행에 넣지 않습니다. 접근 제어된 별도 저장소에 보관하고 중앙 제어면에는 다음 요약만 남깁니다.

- 운영자가 제출한 Dure 커밋 표식, suite와 정책 버전
- 릴리스·배치 프로필과 정렬된 중앙 노드 UUID
- 아티팩트·런타임 다이제스트와 인벤토리 지문
- 작업 부하 크기와 수치 지표
- 통과·실패 상태와 실패 코드

모델 목록을 사람이 관리하는 Markdown 표로 복제하지 않습니다. 모델 레지스트리의 `DRAFT`, `VALIDATED`, `ACTIVE`, `DEPRECATED`, `REVOKED` 상태가 중앙의 진실의 원천입니다. 중앙 추천은 네트워크·NCCL 증적을 정확한 노드 조합의 배치 가능성 판단에 연결하지만, 해당 증적을 자동으로 측정하거나 실제 GPU stage를 배포하지는 않습니다. 적격 증적이 없으면 다중 노드 후보를 실패 안전 방식으로 거부합니다.
